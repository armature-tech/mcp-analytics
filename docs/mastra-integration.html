Integrating @armature-tech/mcp-analytics with Mastra Shape D

Mastra's @mastra/mcp uses its own MCPServer class — different surface from @modelcontextprotocol/sdk's McpServer. The analytics adapter operates at the tool level instead of the server level: it wraps each tool's inputSchema and execute before you hand them to new MCPServer({ tools }).

1. What the integration looks like

One import, one wrap call. The Mastra server stays untouched — same construction call, same OAuth, same transport.

import { MCPServer } from "@mastra/mcp";
import { wrapMastraTools } from "@armature-tech/mcp-analytics/mastra";

new MCPServer({
  id: "my-mcp",
  name: "My MCP",
  version: "0.0.1",
  tools: wrapMastraTools(createMyTools(), {
    armature: { delivery: "await" },
  }),
});

2. What wrapMastraTools transforms

It walks the { name: tool } map you pass to MCPServer, decorates each inputSchema with a telemetry block (so the LLM sees fields for intent / context / frustration_level), and swaps each execute for an instrumented version. Identity of the map shape is preserved — Mastra still sees the tools it expects.

3. Runtime: one tool call end-to-end

When an MCP client calls a wrapped tool, the adapter's execute intercepts the call, extracts the telemetry block from the args, runs the original handler with the cleaned input, and posts a signed batch to Armature after the handler returns (or throws).

4. Where the boundary is

Everything in the dashed Mastra box is unchanged. The adapter sits entirely inside the tool object, so Mastra's protocol handling, OAuth, resources, and HTTP transport keep working as-is. The package never imports @mastra/* at runtime — typing is structural — so any Mastra version with the standard { id, inputSchema, execute } shape works.

5. Carrying session + auth from Mastra's runtime context

Mastra calls execute(inputData, context) — the second arg is whatever the customer's tools already receive (Mastra-specific shape with runtimeContext, request metadata, etc.). Pass a resolveExtra callback to lift sessionId / authInfo / headers off of it into our analytics event.

wrapMastraTools(tools, {
  armature: { delivery: "await" },
  resolveExtra: (mastraContext) => ({
    sessionId: mastraContext?.runtimeContext?.get?.("sessionId"),
    authInfo: mastraContext?.requestContext?.authInfo,
  }),
});

actorId uses the normal config path — the resolver's ctx argument is set to Mastra's second-arg context, so you can pull from runtimeContext there too.

6. Long-lived processes & flushing delivery: background

If the server is a persistent Node process and you want delivery: "background" (default — returns the tool result before the batch is posted), use createMastraAnalytics to keep a handle on flush():

import { createMastraAnalytics } from "@armature-tech/mcp-analytics/mastra";

const analytics = createMastraAnalytics({
  armature: { delivery: "background" },
});

new MCPServer({
  tools: analytics.wrapTools(createMyTools()),
  /* ... */
});

process.on("SIGTERM", () => analytics.flush());

For serverless deployments (per-request lambdas, Vercel, Workers, Mastra hosted on Hono with { serverless: true } — the example service's setup), stay on delivery: "await" and you don't need a flush handle.

7. The full set of exports

wrapMastraTools(tools, opts?)

Convenience: creates an internal recorder, returns the wrapped tool map. Use when you don't need a flush handle.

createMastraAnalytics(opts?)

Returns { recorder, wrapTools, flush }. Use for shared recorders or background delivery.

wrapMastraToolsWithRecorder(tools, recorder, config?, opts?)

Lowest level: bring your own AnalyticsRecorder. Useful if you already construct one for non-Mastra reasons.

MastraAdapterOptions

All of McpAnalyticsConfig plus an optional resolveExtra(mastraContext).
